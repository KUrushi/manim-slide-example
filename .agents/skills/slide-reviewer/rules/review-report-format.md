# レビューレポート出力形式・スコアリング定義

## レポート構造

レポートは以下の6セクションで構成する。順序を変更しないこと。

### 1. レビュー概要
- 対象ファイル、総スライド数、対象聴衆、発表の場、レビュー日
- メタデータとして必ず記載する

### 2. 総合評価
- 3文での全体評価: (1) 良い点、(2) 主な懸念、(3) 最もインパクトの高い改善
- 総合レーティングを付与

### 3. スライド別レビュー
- 各スライドについて:
  - 該当する評価基準のスコアと具体的観察
  - 聴衆体験の要約（1-2文）
- タイトルスライドは「第一印象」基準を含める
- すべてのスライドにコメントが必要なわけではない。特筆すべき点がないスライドは簡潔に記載する

### 4. ナラティブフロー評価
- ストーリーアーク、ペーシング、トランジション、ステアケース効果の各項目
- スライドシーケンスとしての評価

### 5. 優先アクション
- インパクト順に並べた改善提案テーブル
- 各行に: 優先度、対象スライド、課題、改善案、対応スキル

### 6. 強み
- プレゼンテーションの優れた点を具体的に記載
- 良い点を認識することで、改善時に壊してはいけない要素を明確にする

### 7. 聴衆が抱くであろう質問
- プレゼンテーションを見た聴衆が自然に持つであろう質問のリスト
- Q&A 対策として活用可能

## スコアリングルーブリック

### 個別基準のスコアリング

各評価基準（audience-criteria.md の8観点）に対して3段階で評価:

| スコア | 意味 | 基準 |
|--------|------|------|
| **Strong** | 優れている | 観察指標の大部分を満たし、聴衆にとって効果的 |
| **Adequate** | 許容範囲 | 基本的な要件は満たすが、改善の余地がある |
| **Needs Improvement** | 改善必要 | 観察指標を満たさず、聴衆体験を損なう |

### 総合レーティング

全基準のスコア分布に基づいて4段階で評価:

| レーティング | 条件 |
|-------------|------|
| **Strong** | Needs Improvement が0個、Strong が過半数 |
| **Good** | Needs Improvement が1個以下、Adequate 以上が大部分 |
| **Needs Work** | Needs Improvement が2-3個、または重要基準（メッセージ明確性、認知負荷）で Needs Improvement |
| **Major Revision Needed** | Needs Improvement が4個以上、または複数の重要基準で Needs Improvement |

### 重要基準

以下の基準は聴衆体験への影響が特に大きいため、重み付けを高くする:
- **メッセージ明確性** (基準2): プレゼンの存在意義に直結
- **認知負荷** (基準3): 聴衆の理解度に直結
- **ナラティブ一貫性** (基準4): 全体の説得力に直結

## 出力規則

### 言語

- レポート本文は**日本語**で出力する
- 技術用語、コード、ツール名は英語のまま記載可
- 評価スコア（Strong / Adequate / Needs Improvement）は英語のまま使用

### フォーマット

- Markdown 形式で出力
- テーブルは GitHub Flavored Markdown のテーブル構文を使用
- 見出しレベルは `##` から開始（`#` はレポートタイトル用）

### 具体性

- すべての指摘は具体的なスライド番号と観察に基づくこと
- 「改善すべき」だけでなく「どう改善すべきか」を示すこと
- 改善提案には対応するスキル名を明記すること

### 分量

- 総合評価: 3文
- スライド別レビュー: 各スライド1-5行（問題がないスライドは簡潔に）
- 優先アクション: 最大10項目（インパクト順）
- 強み: 3-5項目
- 聴衆の疑問: 3-7項目
